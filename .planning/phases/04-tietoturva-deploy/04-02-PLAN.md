<objective>
Automate production deployment of the Virtuaalitalli application to Altervista using
GitHub Actions and the FTP-Deploy-Action. Replace the manual FTP-upload steps in the
deployment checklist with a workflow that fires on every push to `main` and uploads only
the `public/` directory to the Altervista web root.

Purpose: Eliminate error-prone manual FTP uploads and guarantee that production always
matches the `main` branch, while keeping planning/database/git files out of production.

Output:
- `.github/workflows/deploy.yml` — the CI/CD workflow
- Updated `DEPLOYMENT_CHECKLIST.md` describing the automated flow and one-time Secrets setup
</objective>

  </read_first>
  <action>
    Create `.github/workflows/deploy.yml` — a GitHub Actions workflow named "Deploy to Altervista".

    Trigger (per CONTEXT.md): `on: push: branches: [main]` only. Do NOT add `workflow_dispatch`
    or any other trigger (CONTEXT explicitly defers manual triggers).

    Single job `deploy` running on `ubuntu-latest` with two steps:
    1. `actions/checkout@v4` to check out the repository.
    2. `SamKirkland/FTP-Deploy-Action@v4` configured with:
       - `server: ${{ secrets.FTP_HOST }}`
       - `username: ${{ secrets.FTP_USERNAME }}`
       - `password: ${{ secrets.FTP_PASSWORD }}`
       - `local-dir: ./public/` (note the trailing slash — the action requires it)
       - `server-dir: /` (Altervista web root; CONTEXT notes it may be `/` or `/htdocs/` —
         use `/` and document the alternative in the checklist for the user to confirm)
       - `exclude:` list to belt-and-suspenders block non-deploy paths even though they live
         outside `public/`: `**/.git*`, `**/.git*/**`, `**/.planning/**`, `**/database/**`,
         `**/node_modules/**`, `**/.github/**`

    Because `local-dir` is scoped to `./public/`, `.planning/`, `database/`, and `.github/`
    are already excluded by construction — the `exclude` list is defensive only.

    Credentials MUST come exclusively from `${{ secrets.* }}` — never hardcode host, username,
    or password. Pin the FTP action to `@v4` (CONTEXT: "tai uusin vakaa versio"; v4 is the
    current stable major).

    Do NOT specify a PHP version or run any build/test step — this is a plain PHP app deployed
    as static source files (Claude's Discretion in CONTEXT: PHP version not needed for FTP upload).
  </action>
  <verify>
    <automated>test -f .github/workflows/deploy.yml && grep -q "SamKirkland/FTP-Deploy-Action@v4" .github/workflows/deploy.yml && grep -q "branches:" .github/workflows/deploy.yml && grep -Eq "local-dir:\s*\./public/" .github/workflows/deploy.yml && grep -q "secrets.FTP_HOST" .github/workflows/deploy.yml && grep -q "secrets.FTP_USERNAME" .github/workflows/deploy.yml && grep -q "secrets.FTP_PASSWORD" .github/workflows/deploy.yml && ! grep -q "workflow_dispatch" .github/workflows/deploy.yml && echo OK</automated>
  </verify>
  <acceptance_criteria>

    - Uses `SamKirkland/FTP-Deploy-Action@v4`.
    - `local-dir` is `./public/`; `server-dir` is `/`.
    - FTP_HOST, FTP_USERNAME, FTP_PASSWORD are referenced via `${{ secrets.* }}` and not hardcoded.
    - No build/test/PHP-setup step present.
